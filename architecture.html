SchemaEnricher  ·  src/core/SchemaEnricher.ts
1
Read Raw Schema
MySQLAdapter queries INFORMATION_SCHEMA for all user tables, columns, types, keys, and foreign key relationships. Internal nlsql_* tables are excluded.
MySQLAdapter.getSchema(tables?)
→ TABLE_NAME, COLUMN_NAME, DATA_TYPE,
   IS_NULLABLE, COLUMN_KEY, EXTRA, COLUMN_COMMENT
→ ForeignKey references from KEY_COLUMN_USAGE
2
Sample Column Values
For each column, fetch up to N distinct sample values from the live database. Samples ground the LLM prompt — enum values, ID formats, and date ranges dramatically improve enrichment quality.
addSamplesToTable(rawTable, sampleSize=5)
SELECT DISTINCT {col} FROM {table} LIMIT 5
→ sampleValues: "pending, shipped, delivered"
3
Hash Check · Skip if Fresh
A SHA-256 hash of the table's schema structure (column names, types, nullability, key types) is compared against the stored hash. If the schema hasn't changed, the LLM call is skipped entirely.
EnrichmentStore.hashRawTable(rawTable)
SHA-256( tableName + sorted column definitions )
isEnrichmentFresh() → skip if hashes match
4
LLM Enrichment Call
The schema + samples are sent to the configured LLM. The model returns a structured JSON object with business-friendly names, plain English descriptions, use-cases, and column synonyms.
BaseLLM.enrichTable(rawTable, hash)
→ businessName: "Customer Orders"
→ description: "All purchases made by..."
→ synonyms: ["purchases", "transactions"]
→ columns[].businessLabel, .synonyms,
   .description, .exampleQuestions
5
Persist to MySQL
The enriched description is stored in nlsql_enriched_schema via an upsert. The schema hash is stored alongside it so future runs can detect whether re-enrichment is needed.
EnrichmentStore.save(enrichedTable)
INSERT INTO nlsql_enriched_schema
   (table_name, enriched_json, schema_hash)
ON DUPLICATE KEY UPDATE ...

② Query Phase  ·  every client.query() call
STEP 01
Retrieve Context
Find the most relevant enriched tables for the user's question.
findRelevantTables(query, max=5)
Keyword scoring across:
· table business names
· descriptions & synonyms
· column labels & synonyms
· example questions
Sorted by match count ↓
STEP 02
Generate SQL
The enriched schema context + user question sent to LLM.
BaseLLM.generateSQL(query, tables[])
System prompt rules:
· SELECT only
· table.column notation
· MySQL syntax
· LIMIT 1000 default
Optional few-shot examples
STEP 03
Validate SQL
Two-tier safety check before any query is executed.
SQLValidator.validate(sql)
Tier 1 — Structural:
· Must start with SELECT
· Rejects multi-statements
Tier 2 — Keyword blocklist:
· DROP · DELETE · INSERT
· UPDATE · ALTER · CREATE …
STEP 04
Execute & Return
Validated SQL runs against MySQL. Results + timing returned.
MySQLAdapter.executeQuery(sql, maxRows)
Auto-appends LIMIT if absent

Returns QueryResult:
· sql: string
· results: Record[]
· error: string | null
· timing: { retrievalMs,
  generationMs, validationMs,
  executionMs, totalMs }
Enrichment phase — one-time setup, re-runs only when schema changes
Query phase — every client.query() call at runtime
